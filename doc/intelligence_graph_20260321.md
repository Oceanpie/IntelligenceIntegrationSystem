
# 事件脉络与图谱推演——理论基础与系统设计

## 1. 核心挑战与理论基础
将孤立的情报点串联成线与网，需要克服图数据科学在情报领域的两大核心挑战：
* **超级节点陷阱 (Supernode / Hairball Problem)**：若简单以高频实体（如“美国”、“联合国”）为锚点建图，会导致全网 80% 的节点被强行连接，脉络变成毫无解析价值的乱麻。
* **事件演变与语义漂移 (Event Evolution & Semantic Drift)**：同一主线事件在时间推移中，其物理要素会发生转移（例如：Day 1 巴黎遇袭 -> Day 3 布鲁塞尔搜捕 -> Day 10 海牙审判）。严苛的实体交集会导致主线断裂。

## 2. 脉络推演算法设计：双向深度滚雪球 (Bidirectional N-Hop Snowballing)
系统放弃“全局在线编网”，采用 **基于种子的延迟计算 (Seed-based Lazy Evaluation)**。分析师指定的种子情报（Seed）通常是事件的“爆发点”或“高潮节点”，系统需在**后台线程**中，以该节点为原点，同时向**“过去（溯源）”**和**“未来（追踪）”**两个时间方向展开 N 轮深度迭代查询。

* **2.1 双向时间滑动窗口 (Bidirectional Sliding Windows)**
    * **逆向溯源 (Backward Tracing)**：以 $T_{seed}$ 为起点，时间窗口向过去滑动（如 $T_{-7}, T_{-14}$）。目标是挖掘“酝酿期的微弱信号”、“早期冲突”或“前置因果”。
    * **正向追踪 (Forward Tracing)**：以 $T_{seed}$ 为起点，时间窗口向未来滑动（如 $T_{+7}, T_{+14}$）。目标是追踪“连锁反应”、“各方报复动作”或“后续审判/制裁”。

* **2.2 N 轮异步深度遍历 (Asynchronous N-Depth Iteration)**
    得益于后台线程架构，系统允许对 VectorDB 进行多次查询（图遍历）。
    * **第一轮（1st Degree）**：直接使用 Seed 的向量与实体，在前后滑动窗口内捞出直接相关的一级节点（大概率是同一事件的不同报道或直接后果）。
    * **特征演化与合并 (Context Mutation)**：提取第一轮新发现节点中的**新增罕见实体**，与原 Seed 特征合并，形成新的“查询特征池”。
    * **第 N 轮（N-th Degree）**：带着演化后的特征池，以第一轮边界节点的时间为新基准，继续向过去/未来深度检索。
    * **终止条件 (Stop Conditions)**：
        1. 达到预设的最大迭代深度 $N$（如 3-5 层）。
        2. 候选节点的综合关联得分低于阻断阈值（线索自然中断）。
        3. 触达当前时间点（未来追踪结束）或设定的最长历史界限（如 6 个月前）。

* **2.3 核心打分公式优化 (Absolute Time Penalty)**
    软打分公式中的时间惩罚项需升级为**绝对时间差**，以兼容双向推演：
    $$Score = W_{vec} \times Sim(V_1, V_2) + W_{ent} \times Overlap(E_1, E_2) - W_{time} \times |\Delta T|$$
    *逻辑支撑*：无论是发生在种子前 3 天还是后 3 天，时间维度的衰减惩罚是等效的。结合 N 轮特征池演化，只要 $| \Delta T |$ 在单次跃迁内合理，且向量/实体得分过线，即可建立边（Edge）。

## 3. 数据支撑与存储架构（适配异步机制）
由于推演过程包含 N 次查询与 LLM 评估，耗时可能长达数十秒至数分钟，存储与状态机设计需全面适配异步流。

* **3.1 状态机模型 (State Machine)**
    * `StorylineSnapshot` 文档增加生命周期状态字段 `status`：
      * `QUEUED`: 任务已进入后台队列。
      * `PROCESSING`: 正在进行双向 N 轮检索与图构建。
      * `ANALYZING`: 图谱已生成，正在等待 LLM 进行态势审查与总结。
      * `COMPLETED` / `REJECTED`: 任务完成（通过审查/被系统或人工废弃）。
      * `FAILED`: 过程异常。

* **3.2 文档化图快照 (Document-based Snapshot)**
    * 将包含 Seed、过去节点集、未来节点集以及它们之间的连线（Edges），统一按照发生时间 (`incident_time`) 进行绝对排序，打包为一个完整的 JSON 文档存入数据库。
    * *优势*：前端只需通过轮询（Polling）或 Server-Sent Events (SSE) 监听该文档的 `status`，一旦变为 `COMPLETED`，即可直接拉取这份排序完美的 JSON 渲染时间线，完全感受不到底层的 N 轮复杂图计算。

## 4. 人机协同工作流 (AI-Human Loop) - 【重构版】
1. **触发与遍历**：用户指定 Seed，后台线程启动，双向挖掘出如 20 个相关节点。
2. **时序对齐**：将这 20 个节点严格按照时间线（从最古老的起因，到 Seed，再到最新的后续）排成一个线性序列。
3. **LLM 全局态势审查 (Global Coherence Check)**：
    * 将排序后的序列投喂给 LLM。
    * *Prompt 核心指令*：“这是一条由 AI 在海量数据中双向追溯生成的事件时间线。请审查：其中是否存在因为关键词巧合而被错误插入的‘不连贯杂音节点’？请输出建议剔除的节点 ID，并基于剩余的干净节点，撰写一份包含**‘历史起因 - 核心爆发 - 后续演变’**的结构化态势报告。”
4. **自愈与入库**：后台根据 LLM 的建议剔除脏节点，保存最终干净的快照，状态置为 `COMPLETED`，供分析师在前端看板中消费与最终定级。

## 5. UI/UX 呈现策略
拒绝“蜘蛛网”式的全图渲染，采用双模下钻：
1.  **高维时间轴 (Timeline View)**：主推视图。以垂直时间轴展示事件节点，顶部置顶 LLM 态势总结，专注展示危机演进的起承转合。
2.  **力导向审查图 (Force-Directed Graph)**：辅助视图（基于 ECharts/AntV G6）。用于分析师审计系统的连接逻辑，鼠标悬停连线时，直观展示具体的打分原因（如“向量高度贴合 + 共享罕见实体 X”）。
