
# 事件脉络与图谱推演——理论基础与系统设计

## 1. 核心挑战与理论基础
将孤立的情报点串联成线与网，需要克服图数据科学在情报领域的两大核心挑战：
* **超级节点陷阱 (Supernode / Hairball Problem)**：若简单以高频实体（如“美国”、“联合国”）为锚点建图，会导致全网 80% 的节点被强行连接，脉络变成毫无解析价值的乱麻。
* **事件演变与语义漂移 (Event Evolution & Semantic Drift)**：同一主线事件在时间推移中，其物理要素会发生转移（例如：Day 1 巴黎遇袭 -> Day 3 布鲁塞尔搜捕 -> Day 10 海牙审判）。严苛的实体交集会导致主线断裂。

## 2. 脉络推演算法设计
系统放弃“全局在线编网”，采用 **延迟计算 (Lazy Evaluation)**。即指定一篇高价值情报为种子（Seed），在查询时向外动态推演图谱。

* **核心算法：软打分机制 (Hybrid Weighted Scoring)**
    放弃严苛的五要素完全交集，将向量（事件本质）与实体（参与者/地点）作为互补权重。计算公式如下：
    $$Score = W_{vec} \times Sim(V_1, V_2) + W_{ent} \times Overlap(E_1, E_2) - W_{time} \times \Delta T$$
    *逻辑支撑*：若实体全变但事件相同（向量相似度极高），或事件发生转折但核心罕见人物高度重合，总分依然能跨越阈值维持连接。
* **防蔓延机制**：
    * **TF-IDF 实体降权**：剔除高频词，仅依靠极其罕见的核心实体计算 $Overlap$。
    * **时间衰减窗口**：引入方向性的滑动时间窗，随时间跨度增加，强行提高关联阈值，防止脉络无限蔓延。
* **滚雪球追踪 (Snowball Chaining)**：允许系统捕捉中间节点（1st Degree）的新增罕见实体，借此向下接力追踪，完美解决物理要素的“时空转移”。

## 3. 数据支撑与存储架构
放弃传统全局图数据库（如 Neo4j），以防试错连接污染全局底座知识。

* **文档化图快照 (Document-based Snapshot)**：
    * 将单次动态推演生成的图（包含 Seed ID, Nodes, Edges 的具体得分与连接原因）整体打包为一个 JSON 文档（`StorylineSnapshot`）。
    * 存入文档型/关系型数据库（如 MongoDB 或 PostgreSQL JSONB）。
    * *优势*：极速的读写性能，支持图谱的随时丢弃与版本迭代。

## 4. 人机协同工作流 (AI-Human Loop)
* **图谱生成**：系统基于算法捞出相关节点与边，生成快照。
* **LLM 态势审查**：将快照内的节点按时间排序投喂给大模型。由 LLM 担任裁判，审查逻辑连贯性，挑出由“重名/巧合”引发的杂音，并输出整体评分（1-10）与态势演进总结。
* **生命周期管理**：若 LLM 评分过低，自动将该快照置为 `REJECTED`；合格者流转至前端供分析师确认（`human_rating`）。

## 5. UI/UX 呈现策略
拒绝“蜘蛛网”式的全图渲染，采用双模下钻：
1.  **高维时间轴 (Timeline View)**：主推视图。以垂直时间轴展示事件节点，顶部置顶 LLM 态势总结，专注展示危机演进的起承转合。
2.  **力导向审查图 (Force-Directed Graph)**：辅助视图（基于 ECharts/AntV G6）。用于分析师审计系统的连接逻辑，鼠标悬停连线时，直观展示具体的打分原因（如“向量高度贴合 + 共享罕见实体 X”）。
